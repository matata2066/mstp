# MSTP - Maker/Checker 工作流设计

## 1. 概述

Maker/Checker（制单/审核）是银行核心的风控机制，确保所有关键数据变更经过双人复核。MSTP 系统中所有映射关系的维护操作均需遵循 Maker/Checker 流程。

### 1.1 设计原则

- **职责分离**：同一人不能同时作为 Maker 和 Checker 参与同一笔操作
- **完整追溯**：所有操作保留变更前后快照，可追溯
- **不可绕过**：所有数据变更必须经过审批，无直接修改通道
- **状态明确**：审批记录状态清晰，待审批/已通过/已拒绝

## 2. 角色定义

| 角色 | 职责 | 约束 |
|------|------|------|
| Maker（制单人） | 发起数据变更申请 | 不能审批自己提交的申请 |
| Checker（审核人） | 审核并决定通过或拒绝 | 不能审核自己提交的申请 |
| Admin（管理员） | 拥有 Maker + Checker 权限 | 生产环境建议 Admin 仅用于紧急情况 |

## 3. 工作流状态机

### 3.1 审批流程状态

```
                    Maker 发起
                        │
                        ▼
                  ┌──────────┐
                  │ PENDING  │  待审批
                  └────┬─────┘
                       │
              ┌────────┼────────┐
              │                 │
         Checker 通过      Checker 拒绝
              │                 │
              ▼                 ▼
        ┌──────────┐     ┌──────────┐
        │ APPROVED │     │ REJECTED │
        └────┬─────┘     └────┬─────┘
             │                 │
             ▼                 ▼
      生效到业务表          通知 Maker
```

### 3.2 状态说明

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| PENDING | 等待审核 | Checker 可通过/拒绝；Maker 可撤回 |
| APPROVED | 审核通过 | 变更已生效，不可撤销 |
| REJECTED | 审核拒绝 | Maker 可修改后重新提交 |
| WITHDRAWN | 已撤回 | Maker 撤回的申请，可重新编辑提交 |

## 4. 操作类型与审批流程

### 4.1 新增操作（CREATE）

```
Maker                              Checker
  │                                   │
  │  1. 填写新增表单                    │
  │  2. 提交审批                        │
  │     ── 保存至审批表 ──►             │
  │     (状态: PENDING)                │
  │                                   │
  │                          3. 查看待审批列表
  │                          4. 打开审批详情
  │                          5. 审核变更内容
  │                                   │
  │                     ┌── 通过 ──┐   │
  │                     │          │   │
  │                     ▼          ▼   │
  │              写入业务表    记录拒绝  │
  │              (状态:APPROVED) (REJECTED)
  │                                   │
```

**审批通过后动作：**
- 将审批表中的数据写入对应业务表
- 更新审批记录状态为 APPROVED
- 记录审计日志

### 4.2 修改操作（UPDATE）

```
Maker                              Checker
  │                                   │
  │  1. 选择已有记录                    │
  │  2. 修改字段值                      │
  │  3. 提交审批                        │
  │     ── 保存变更快照 ──►             │
  │     (变更前 + 变更后)              │
  │     (状态: PENDING)                │
  │                                   │
  │                          4. 审核变更对比
  │                             显示: 旧值 → 新值
  │                                   │
  │                     ┌── 通过 ──┐   │
  │                     │          │   │
  │                     ▼          ▼   │
  │              更新业务表    记录拒绝  │
  │              (状态:APPROVED) (REJECTED)
  │                                   │
```

**变更快照格式（OPERATION_DATA）：**

```json
{
  "operationType": "UPDATE",
  "before": {
    "accountNo": "6225880123456789",
    "accountName": "张三",
    "currency": "CNY",
    "status": "ACTIVE"
  },
  "after": {
    "accountNo": "6225880123456789",
    "accountName": "张三丰",
    "currency": "CNY",
    "status": "ACTIVE"
  },
  "changedFields": ["accountName"]
}
```

### 4.3 删除操作（DELETE）

- 逻辑删除，将 IS_DELETED 标记为 1
- 审批通过后执行逻辑删除
- 删除前需确认该记录无关联交易

### 4.4 授权操作（AUTHORIZE）

- 付款账号授权为特殊操作
- 新增付款账号配置时自动触发授权检查
- 如未授权，提示 Maker 发起授权
- 授权操作同样需要 Checker 审批

## 5. 业务场景审批规则

### 5.1 收款账号映射审批

| 操作 | 需审批 | 审批内容 | 通过后动作 |
|------|--------|----------|-----------|
| 新增 | 是 | 账号、名称、币种、类型 | 写入 MSTP_ACCOUNT_MAPPING |
| 修改 | 是 | 变更前后对比 | 更新 MSTP_ACCOUNT_MAPPING |
| 停用 | 是 | 停用原因 | 更新 STATUS=INACTIVE |
| 删除 | 是 | 删除原因 | 逻辑删除 |

### 5.2 行号映射审批

| 操作 | 需审批 | 审批内容 | 通过后动作 |
|------|--------|----------|-----------|
| 新增 | 是 | 行号、行名、类型 | 写入 MSTP_BANK_MAPPING |
| 修改 | 是 | 变更前后对比 | 更新 MSTP_BANK_MAPPING |
| 停用 | 是 | 停用原因 | 更新 STATUS=INACTIVE |
| 删除 | 是 | 删除原因 | 逻辑删除 |

### 5.3 付款账号配置审批

| 操作 | 需审批 | 审批内容 | 通过后动作 |
|------|--------|----------|-----------|
| 新增 | 是 | 渠道、付款信息、币种 | 写入 MSTP_PAYER_ACCOUNT_CONFIG |
| 修改 | 是 | 变更前后对比 | 更新 MSTP_PAYER_ACCOUNT_CONFIG |
| 授权 | 是 | 授权账号、渠道 | 调用下游授权API + 更新授权状态 |
| 停用 | 是 | 停用原因 | 更新 STATUS=INACTIVE |

## 6. 并发控制

### 6.1 同一记录并发修改

- 乐观锁：业务表增加 VERSION 字段
- Maker A 修改记录时，若 Maker B 已提交修改且待审批，提示"该记录有待审批的变更，请稍后操作"
- 审批通过时校验版本号，版本不一致则审批失败

### 6.2 同一审批单并发审批

- 数据库行锁：审批时对审批记录加行锁
- 第一个审批操作成功后，后续审批操作提示"该审批单已处理"

## 7. 通知机制

### 7.1 通知场景

| 场景 | 通知对象 | 通知方式 |
|------|----------|----------|
| 新审批单提交 | Checker 角色 | 站内消息 + 邮件（可选） |
| 审批通过 | Maker | 站内消息 |
| 审批拒绝 | Maker | 站内消息 + 邮件（可选） |
| 授权结果 | Maker | 站内消息 |

### 7.2 站内消息设计

- 页面顶部显示未读消息数量
- 点击可查看消息列表
- 消息关联审批单号，点击可跳转至审批详情

## 8. 异常处理

### 8.1 审批通过后业务写入失败

- 审批记录标记为 APPROVED
- 业务数据写入失败时，审批记录增加异常标记
- 自动重试3次，仍失败则告警，需人工介入
- 提供手动重试功能

### 8.2 审批超时

- 审批单超过48小时未处理，自动提醒 Checker
- 审批单超过7天未处理，升级通知 Admin
- 不自动关闭审批单，需人工处理

## 9. 审计与合规

### 9.1 审计要求

- 所有审批操作记录审计日志
- 审计日志包含：操作人、操作时间、操作类型、变更内容
- 审计日志不可修改、不可删除
- 支持按时间范围、操作人、业务类型查询审计记录

### 9.2 合规检查

- 系统定期检查是否存在 Maker/Checker 同一人的情况
- 检查是否存在长期未处理的审批单
- 检查是否存在审批流程异常（如秒批等可疑行为）
- 检查结果生成合规报告
